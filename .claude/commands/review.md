以下の観点で $ARGUMENTS を全ファイルにわたってレビューしてください。

- 下記は「原則」と「代表的な赤旗（例）」のセット。例に一致しない実装でも、同等のリスクがあれば同様に指摘する。
- 文字列マッチではなく、入力経路・データフロー・失敗時挙動を含めて総合判断する。

## 指摘の優先度
- **Must**: セキュリティ問題、クラッシュ、データ破壊、明確な回帰は修正必須
- **Should**: 将来バグ化しやすい型の曖昧さ・設計の脆さは原則修正
- **May**: スタイル/好みの差は別チケット化も可

## 見逃し厳禁の赤旗
- XSS / CSS インジェクション / パストラバーサル / 任意コマンド実行の可能性は、実装の都合に関わらず最優先で検出・指摘する
- 入力未検証のまま DOM・CSS・ファイルパス・シェル引数へ到達する経路は、具体パターンに一致しなくても Must 扱いで止める
- 認証・権限・機密情報（トークン、環境変数、ログ出力）に関わる漏えいリスクは Must 扱いで確認する

## セキュリティ
- 外部入力は常に未信頼として扱い、DOM・CSS・ファイルパス・コマンド引数に到達する前に検証/無害化する
- 例: `innerHTML` への未サニタイズ注入、`background` への危険値、`path.resolve(cwd, userInput)` の任意書き込み
- CLI は引数境界と型を必ず検証し、`args[++i]` 越境や `parseInt` の NaN 伝播を防ぐ

## バグ
- 失敗しうる処理（パース、I/O、外部依存）は失敗前提で扱い、例外時に安全に失敗させる
- ライフサイクルを持つ資源（`BroadcastChannel` / `setInterval` / `addEventListener` など）は必ず cleanup する
- 実行環境差（SSG/CSR）を踏まえ、`Astro.url.searchParams` など環境依存値の扱いを誤らない
- 数値変換や配列アクセス後は `NaN` / `undefined` を検証してから後続に渡す
- 例示にない不具合（状態不整合、境界値、非同期競合、初期化順、タイムゾーン/ロケール依存）も同じ優先度基準で評価する

## 型安全性
- 型注釈よりランタイム事実を優先し、未検証データは型ガードで絞り込む
- 二段キャスト（`as unknown as T`）などの強引な型合わせは避ける
- オプション値や外部入力は存在/型を確認してから `join()` / `resolve()` などに渡す

## パフォーマンス
- 高頻度経路の無駄な再計算・再生成を避ける（例: ループ内 `new RegExp(...)`）
- 高コスト資源（ブラウザ/プロセス/接続）は使い回しを優先し、起動回数を最小化する
- DOM イベントは可能な限りデリゲーションで束ねる

## 保守性
- 重複ロジックや分散設定を避け、単一責務・単一定義を保つ
- 長大関数は責務で分割し、読みやすさと変更容易性を維持する
- 危険設定（例: `allowDangerousHtml: true`）には意図と安全策をコメントで明示する

## テスト
- 変更には対応する unit (`packages/astlide/tests/`) または e2e (`e2e/`) の追加・更新を伴うことを確認する
- バグ修正には再現するテストを必ず付ける

## 公開API互換性
- `package.json` の `exports` マップに含まれる型・インターフェースの破壊的変更は明示的なメジャーバンプが必要
- ユーザー向けの `slideSchema` や Astro Integration の設定型が壊れていないかを確認する

## アクセシビリティ
- 支援技術の利用者が情報を失わないことを優先する
- `aria-hidden="true"` は段階表示コンテンツに使わない
- `aria-label` は同種要素間で識別可能にし、`prefers-reduced-motion` で動きを抑制する
